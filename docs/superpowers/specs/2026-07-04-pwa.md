# PWA Design

## Summary

Turn Archi Online into a fully featured Progressive Web App: installable,
fully offline-capable, and OS-integrated. The app is already an ideal PWA
candidate â€” no backend, models live in `.archimate` files and IndexedDB
autosave, and every runtime asset (Monaco, ELK, fonts) is self-hosted.

## Goals

- Installable with proper icons, name, and theme colors on desktop and mobile.
- Full offline support: the entire build (~16 MB, ~97 hashed assets, including
  Monaco workers, ELK, the example models, and `popout.html`) is precached, so
  everything works offline from the first visit.
- OS file handling: double-clicking a `.archimate` file opens it in the
  installed app, with the file handle retained so silent Ctrl+S re-save works.
- App shortcuts: "New model" and "Open model file" jump actions on the
  installed app icon.
- Web Share Target: sharing an `.archimate` file to the installed app opens it.
- In-app update notification: a toast offers "Reload" when a new deployment is
  available, instead of silently serving a stale version.

## Non-Goals

- No `window-controls-overlay` display mode in v1. It requires the toolbar to
  reserve `titlebar-area-*` env() space; noted as a follow-up.
- No background sync, push notifications, or periodic sync.
- No offline support for gist/raw share links â€” those are cross-origin GitHub
  fetches and stay network-only. Inline (`#m=`) links work offline because the
  model travels in the URL fragment.
- No change to hosting: the app stays root-hosted (`base: '/'`).

## Approach

`vite-plugin-pwa` (peer-supports Vite 6) with `strategies: 'injectManifest'`
and a small hand-written `src/sw.ts`.

`injectManifest` over `generateSW` is decisive because of the share target:
receiving a shared file requires the service worker to handle a
`POST /share-target` navigation â€” read `formData()`, stash the file in Cache
Storage, and respond with a 303 redirect into the app. `generateSW`'s
`runtimeCaching` cannot express custom POST response logic. Precaching is
identical to `generateSW` via `precacheAndRoute(self.__WB_MANIFEST)`.

Service worker routing:

- Precache the full build manifest (`globPatterns` include js, css, html, png,
  svg, ttf, webmanifest, and `.archimate` examples;
  `maximumFileSizeToCacheInBytes` raised to 8 MB for the ~7 MB Monaco
  `ts.worker`).
- SPA navigations resolve to the precached `index.html` via a
  `NavigationRoute`, so `?mode=viewer` share URLs load offline. `popout.html`
  and `/share-target` are denylisted â€” the dockview popout window must be
  served as itself, never the SPA shell.
- No route matches GitHub requests (`api.github.com`,
  `gist.githubusercontent.com`, `raw.githubusercontent.com`), so they fall
  through to the network untouched. Offline they fail into the existing
  `ViewerShell` error state.

Update flow: `registerType: 'prompt'` â€” a new service worker waits until the
user confirms. `UpdatePrompt.tsx` (mounted in `main.tsx` so it covers editor
and viewer modes; `AppDialogHost` only exists in editor mode) shows a toast.
Reload first flushes the debounced autosave and bypasses the `beforeunload`
dirty guard once, so no work is lost and no double prompt appears.

File handling: manifest `file_handlers` for `.archimate` plus a
`window.launchQueue` consumer registered before render. The consumer waits for
the editor runtime boot (autosave restore) to finish, then asks to discard
unsaved work and opens the launched file through
`openModelFromHandle()`, which keeps the `FileSystemFileHandle` for silent
re-save.

## Platform Support Matrix

| Capability | Chromium desktop | Chromium Android | Safari/iOS | Firefox |
| --- | --- | --- | --- | --- |
| Install + offline | yes | yes | yes (A2HS) | no install UI |
| `file_handlers` / `launchQueue` | yes | no | no | no |
| Shortcuts | yes | yes | no | no |
| Share target (files) | Windows Edge | yes | no | no |
| Update prompt | yes | yes | yes | yes |

All Chromium-only surfaces are feature-detected and degrade to no-ops.

## Security / Privacy Notes

- The service worker never caches GitHub API responses or tokens.
- The share-target inbox lives in Cache Storage under a dedicated cache name
  and is deleted as soon as the app consumes it.
